How It Works
Step 1 — Dual Indexing
At upload time, documents are indexed twice: as embedding vectors (dense) for semantic search, and as a TF-IDF matrix (sparse) for keyword search. Both indexes are maintained simultaneously in memory and queried in parallel on every request.
Step 2 — Parallel Retrieval
The same query runs through both systems at once. Dense retrieval finds semantically similar chunks (good for paraphrasing, synonyms, and conceptual matches). Sparse retrieval finds exact keyword matches (good for names, codes, product IDs, and technical terms).
Step 3 — RRF Fusion
Reciprocal Rank Fusion combines both ranked lists without needing learned weights. Each chunk gets a score of 1/(k+rank) from each list. Scores are summed across lists. k=60 is the standard constant that prevents very high scores for top-ranked documents.
Step 4 — LLM Generation
The top fused chunks are passed to the LLM. Each chunk carries a source badge (vector, BM25, or both) indicating which retrieval method found it — this transparency helps you understand why each piece of context was included.

Step 4 — RRF Fusion
RRF scores each chunk as 1/(60+rank) from each list, then sums across lists. Chunks that appear highly in both lists float to the top. RRF does not operate on vectors — it only uses rank positions.

Step 5 — Cross-Encoder Reranking
Used in Advanced RAG — not in this Hybrid RAG implementation, but shown here so you understand the full evolution
A Cross-Encoder takes the user query paired with each candidate chunk and scores their relevance jointly. Unlike embeddings (which encode query and chunk independently), the Cross-Encoder reads both together — letting transformer attention compare their tokens directly.

When to Use It
  • Queries mix semantic questions and exact keyword lookups
  • Documents contain technical terms, product codes, or proper nouns
  • You want better recall than pure vector search
  • You want a practical upgrade over Naive RAG without full Advanced complexity
When Not to Use It
  • Documents are purely narrative with no specific keywords to match
  • Latency is critical — running two searches (dense + sparse) adds overhead compared to a single search
  • Infrastructure only supports one index type
Strengths
  • Best of both worlds — semantic coverage plus keyword precision
  • Simple fusion via RRF — no learned weights or training required
  • Noticeably better recall than naive vector-only search
Limitations
  • Double the indexing overhead — two structures to build and maintain
  • Two query paths to keep in sync
  • Still no reranking — RRF is better than one signal but not as precise as a cross-encoder
Key Concepts
Dense Retrieval vector search · semantic
Uses embeddings to find chunks that are semantically similar to the query, even if they use different words. This app uses all-MiniLM-L6-v2 and a NumPy dot product — the same model used across all RAG strategies.
Sparse Retrieval TF-IDF BM25 · keyword
Uses term frequency and inverse document frequency to score chunks that contain the exact words from the query. Excellent for technical terms, proper names, codes, or any string where the exact wording matters. This app uses scikit-learn's TfidfVectorizer.
Reciprocal Rank Fusion formula: 1/(k+rank) · k=60
A score-free algorithm for merging ranked lists from different retrieval systems. Each document in each list receives a score of 1/(60+rank). Scores are summed. Documents that rank highly in multiple lists bubble to the top. No training or learned weights are needed.
Implementation in This App
This app uses NumPy dot product for dense search and sklearn TF-IDF for sparse search. Both indexes are built at upload time via rebuild_indexes() and stored in memory. The reciprocal_rank_fusion() utility handles the merge.
