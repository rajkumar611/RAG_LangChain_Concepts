STRATEGY 5 OF 5 Graph RAG Multi-Hop Reasoning

Instead of retrieving isolated text chunks, Graph RAG extracts entities and relationships from documents and stores them as a knowledge graph. At query time, the system traverses edges between entities — following chains of connections across multiple documents — to answer questions that no single chunk could resolve alone.

How It Works
Step 1 — Entity & Relationship Extraction
An LLM reads each document and extracts entities (people, teams, technologies, projects, locations, concepts) and the relationships between them (works_at, manages, uses, depends_on, located_in). This is the most expensive step — it requires one LLM call per document — but it is a one-time cost done during ingestion, not at query time. The quality of the entire graph depends on this step: missed entities or mislabelled relationships propagate silently into wrong answers.
Step 2 — Knowledge Graph Construction
Extracted entities become nodes with attributes (name, role, version, date). Extracted relationships become directed edges between those nodes. The graph is stored in a graph database such as Neo4j, ArangoDB, or Amazon Neptune, which is purpose-built for fast multi-hop traversal. Related entities are also grouped into communities — clusters of tightly connected nodes — which can be summarised and retrieved as a unit for broader questions.
Step 3 — Hybrid Storage (Graph DB + Vector DB)
Graph RAG uses two databases simultaneously. The graph database stores the explicit structure: nodes, edges, and attributes. A vector database stores semantic embeddings of each entity and relationship description. This pairing enables both structural traversal ("follow the manages edge from Sarah") and semantic lookup ("find entities related to 'container orchestration'") — covering questions the other database alone cannot answer.

Step 4 — Query Time: Entity Matching
The user's question is parsed to identify key entities and concepts. Vector search finds the semantically closest nodes in the graph as entry points. These become the starting nodes for traversal. This step determines which part of the graph the traversal begins from — choosing the wrong entry node means the traversal follows the wrong chain entirely.
Step 5 — Multi-Hop Graph Traversal
The system follows edges from the starting node, collecting each linked entity along the path. Each edge crossing is called a hop. A 3-hop traversal can connect an employee → their team → the team's technology → the technology's vulnerability. Crucially, this information may live in four separate documents — the graph stitches them together into one coherent reasoning chain. The traversal path itself can be shown to the user as a transparent audit trail for the answer.
Step 6 — LLM Generation
The collected entities, their attributes, and the traversal path are passed to the LLM as context. Because the context is structured and relationship-constrained — not loose text chunks — the LLM is less likely to hallucinate connections that do not exist in the data. The answer can include a citation showing the exact graph path that produced it.
What Is a Hop?

A hop is one step through an edge in the knowledge graph — like jumping from stone to stone across a river. Each jump follows a named relationship between two entities. Questions that require multiple jumps are called multi-hop questions.

Each node comes from a different document: Alex is in the Team Directory, Kubernetes is in the Tech Stack doc, the CVE is in the Security Bulletin. Graph RAG connects them automatically. Naive or Advanced RAG would need all this information to appear together in one chunk — it never does.

When to Use It
  • More than 10 related documents with scattered relationships
  • Questions that require reasoning across multiple documents at once
  • Explainability is required — users need to see the reasoning chain
  • Enterprise systems: HR, compliance, security, project tracking
  • Questions about who, where, which team, what technology — relationship-heavy
When Not to Use It
  • Working with a single document — overhead is not justified
  • Questions are conceptual or fuzzy ("what are best practices?")
  • Documents update frequently — graph re-extraction is expensive
  • Budget is tight — entity extraction requires many LLM calls
  • Team lacks expertise to define the right relationship schema
Strengths
  • Multi-hop reasoning is a native graph operation — not an approximation
  • Explicit relationships reduce hallucination — the model can't invent connections that aren't in the graph
  • Fully explainable — every answer comes with a traceable relationship path
  • Efficient at scale — build once, traverse cheaply across thousands of documents
  • Handles ambiguous entity names via attribute context (two people named "Alex" are distinct nodes)
Limitations
  • Expensive to build — every document requires LLM-powered extraction
  • Garbage in, garbage out — bad extraction produces a graph that answers confidently but incorrectly
  • Missed relationships are silent — the graph doesn't warn you about what it doesn't know
  • Poor for fuzzy or conceptual questions — graphs store facts, not intuitions
  • Requires domain expertise to define the right entity types and relationship schema
Key Concepts
Node
An entity in the knowledge graph — a person, company, technology, project, concept, or location. Each node has attributes: [Alex Kumar] {title: "Engineer", joined: 2022}. Nodes are the "things" the system knows about.
Edge
A directed, named relationship between two nodes. Examples: works_at, manages, uses, has_vulnerability, depends_on. Edges can also carry attributes, such as {since: 2021} on a works_at edge.
Hop
One step through an edge during graph traversal. A 1-hop query follows a single edge (direct relationship). A 3-hop query follows three edges in sequence, crossing three entity boundaries. Enterprise questions commonly require 3–5 hops. The depth of traversal is the main way to control cost vs. completeness.
Entity Extraction
The process of reading documents with an LLM and identifying all entities and relationships within them. This is where Graph RAG's quality is determined. A good extraction prompt is domain-specific — it tells the LLM what types of entities matter (e.g., "people", "teams", "CVEs") and what relationships to look for. Poor prompt design here causes the entire graph to be incomplete or incorrect.
Community Detection
An algorithm (such as Louvain or Leiden) that groups densely connected nodes into communities. All nodes in a community tend to be about the same topic. Summaries of each community can be stored as embeddings and retrieved for broad, thematic questions — a "zoomed out" view of the graph. This enables Graph RAG to answer both specific multi-hop questions and general thematic questions.
Graph DB vs. Vector DB Graph RAG uses both
A graph database (Neo4j, ArangoDB, Neptune) stores nodes and edges and is optimised for traversal queries: "find all entities reachable from node X within 3 hops." A vector database (Pinecone, Chroma, Weaviate) stores embeddings and is optimised for similarity search: "find entities semantically close to this query." Graph RAG needs both — graph DB for structural traversal, vector DB for the semantic entry point into the graph.
This App's Implementation Simplified Sequential Graph
The Graph RAG endpoint in this app uses a simplified approach to demonstrate the graph traversal concept without requiring a graph database. Document chunks become nodes, and edges connect each chunk to its sequential neighbours (the chunk before and after it in the document). A seed retrieval via vector search finds the most relevant chunk, then BFS (Breadth-First Search) expands to neighbouring chunks to collect surrounding context. This is built with networkx in Python. In production, you would replace this with entity extraction → Neo4j/ArangoDB → relationship-aware traversal.
Why not a real Graph DB?
Two reasons: infrastructure and cost.
A real Graph DB (Neo4j, ArangoDB) requires a separately running database server — another process, connection pooling, authentication, schema design. This app has zero external dependencies — no database, no auth, everything lives in Python memory and resets on restart. Adding Neo4j would mean: install Neo4j → run it as a service → manage connections → handle failures. That's a significant infrastructure commitment for a demo.
Entity extraction also requires LLM calls per chunk at ingestion time:
Every chunk → LLM call → "Extract entities and relationships" → write to Neo4j
For a 50-page document with 100 chunks → 100 LLM API calls just to index one document. Slow, expensive, and adds API key dependency to ingestion — not suitable for a lightweight demo.
What we use instead — networkx
networkx is a pure Python in-memory graph library. No server, no setup, no cost. The app sidesteps entity extraction entirely by deriving graph structure from chunk order:
Chunk 0 ── Chunk 1 ── Chunk 2 ── Chunk 3 ── Chunk 4
Each chunk is a node. Edges connect sequential neighbours. No LLM needed to build this — it's just array indexing. BFS then expands from the seed chunk to pull in neighbours.

What is "Seed Retrieval"?
Seed retrieval is the first step — finding the starting point in the graph before any traversal begins. Think of being dropped into a city and needing to find a specific neighbourhood: before you can navigate the streets, you need to know which street to start on. Finding that starting street = seed retrieval.
User Query: "Which teams are affected by the vulnerability?"
        │
        ▼
   Vector search (cosine similarity)
        │
        ▼
   Returns the 2–3 most relevant chunks
        │
        └── These are the SEED nodes
                │
                ▼
           BFS expands outward → pulls in neighbouring chunks
Without seed retrieval, you'd have to traverse the entire graph from every node — computationally pointless. The seed narrows it down: "start traversal from here, not from everywhere."
So the full flow is: ① Seed retrieval — vector search finds the entry point  →  ② Graph traversal — BFS expands outward  →  ③ Re-score — rank everything collected  →  ④ LLM — answer from combined context.
