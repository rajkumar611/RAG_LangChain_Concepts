Graph RAG

Documents are nodes in a sequential knowledge graph. Retrieval seeds on the best matching chunk, then expands across graph edges using BFS to pull in related context — ideal for multi-hop and relational questions.

→

How It Works
Step 1 — Graph Construction
At upload time, chunks are arranged as nodes in a sequential graph where each node connects to its immediate neighbours — the chunks before and after it in the document. This preserves the document's natural flow and structure without requiring entity extraction or manual annotation.
Step 2 — Seed Retrieval
The user's query is embedded and the top-K most similar chunks are found via vector search. These are the seed nodes — the starting points for graph traversal. Seed selection uses cosine similarity, the same as Naive RAG.
Step 3 — BFS Expansion
Starting from the seed nodes, the system traverses the graph via Breadth-First Search to collect neighbouring chunks. This brings in surrounding context — the passage before and after each seed — that may not have matched the query directly but provides critical continuity.
Step 4 — Re-scoring
All collected chunks (seeds and BFS neighbours) are re-scored against the original query using cosine similarity. The highest-scoring subset is selected. Re-scoring ensures that BFS-expanded context that turns out to be irrelevant is ranked below seed nodes that directly match.
Step 5 — LLM Generation
The final scored chunks are passed to the LLM. The result display shows which nodes were seed nodes (green) and which were BFS-expanded neighbours (grey), giving you visibility into how the graph traversal contributed to the answer.
When to Use It
  • Questions requiring context around the retrieved passage — what came before or after
  • Documents with sequential dependencies — reports, manuals, narratives
  • Multi-hop reasoning across related, adjacent sections
When Not to Use It
  • Documents are unordered collections — FAQs, notes, bullet lists
  • Each chunk is fully self-contained — no benefit to reading neighbours
  • You need the fastest possible retrieval with minimal overhead
Strengths
  • Retrieves surrounding context automatically — no prompt engineering needed
  • Good for documents where meaning spans multiple adjacent passages
  • No extra metadata needed — graph is built from chunk order alone
Limitations
  • Sequential graph is a simplification — real knowledge graphs use entity extraction
  • BFS expansion can bring in irrelevant neighbours near the seed
  • More expensive than naive retrieval — extra graph traversal and re-scoring
Key Concepts
Sequential Graph nodes=chunks · edges=adjacency
Each document chunk is a node. Edges connect sequential neighbours — chunk 5 connects to chunks 4 and 6. Built with NetworkX. This graph captures the document's linear narrative structure without requiring entity extraction.
BFS Expansion breadth-first search from seed nodes
Starting from the seed nodes identified by vector search, Breadth-First Search expands outward to collect all neighbours within a certain hop distance. This brings in the surrounding passage that may not have matched the query directly but provides essential context.
Re-scoring re-rank expanded set by cosine similarity
After BFS expansion, all candidate chunks — seeds and neighbours — are re-ranked against the original query. This ensures that irrelevant BFS-expanded chunks don't crowd out the most relevant seed nodes in the final context window.
NetworkX
This app uses the networkx Python library to build and traverse the sequential document graph. The global variable G holds the graph, rebuilt on every upload via rebuild_indexes().
