How It Works
Step 1 — Document Ingestion, Cleaning & Smart Chunking
Raw data arrives from multiple sources. A parsing layer extracts clean text before chunking begins — headers, footers, encoding noise, and boilerplate are removed first. Unlike fixed-size splitting, Advanced RAG uses strategies that respect meaning boundaries.
Input Sources
  • PDFs (OCR if scanned)
  • Web pages
  • APIs & internal docs
  • Tables & images extracted separately
Cleaning Steps
  • Remove headers, footers, boilerplate
  • Fix encoding issues
  • Normalise whitespace & noise
  • Extract structured text
Sentence-Aware Chunking
Splits only at sentence boundaries — never in the middle of a sentence. Preserves grammatical completeness.
Semantic Chunking
Groups consecutive sentences by embedding similarity. A new chunk begins when topic drift is detected — keeps related ideas together.
Recursive Chunking (Hierarchical)
Splits top-down: section → paragraph → sentence. Small child chunks are stored for precise retrieval; parent chunks provide richer LLM context.
Input text (paragraph):
"Self-attention computes relationships between all tokens in a sequence. Each token attends to every other token, producing a weighted sum of value vectors."

Sentence-aware output (2 chunks):
Chunk A: "Self-attention computes relationships between all tokens in a sequence."
Chunk B: "Each token attends to every other token, producing a weighted sum of value vectors."
Step 2 — Metadata Enrichment & Dual Storage
Every chunk is tagged with structured metadata at ingestion time. This metadata powers filtering, ranking boosts, and source traceability. Each chunk is then stored in two completely separate systems — a Vector Index (Dense) for semantic similarity search and a Keyword Index (Sparse) for exact keyword matching.
Chunk metadata example:
{
  "source": "attention_paper.pdf",
  "section": "3.2 Self-Attention",
  "topic": "Transformers",
  "year": 2017,
  "chunk_id": 42
}
Vector Index (Dense)
Stores the embedding vector for each chunk. Used for cosine / semantic similarity search.
"self attention computes..."
→ [0.21, -0.88, 0.77, ...]
→ stored in Pinecone / Milvus / NumPy
Vector Index (Dense) is the data structure concept — a searchable store of embedding vectors. A Vector DB (Pinecone, Chroma, FAISS, Milvus) is a production database product that hosts this index with persistence, scalability, and fast ANN search. Same idea, different abstraction level. In this app the index is a plain NumPy array in memory.
Keyword Index (Sparse)
Stores a term → chunk mapping (inverted index structure). Lookup is direct — no vector comparison ever happens here.
"attention" → [chunk 3, chunk 17, chunk 42]
"transformer" → [chunk 1, chunk 42]
→ stored in Elasticsearch / TF-IDF matrix
At query time, metadata enables pre-filtering (only search chunks from a specific year or section), ranking boosts (prefer recent sources), and traceability — every answer can be traced to an exact source and section.
Optional Extension — Multi-Vector Indexing
The intro at the top of this page mentions multi-vector indexing. This is an optional extension to dual storage: instead of one dense embedding per chunk, store multiple embeddings — each capturing a different representation of the same chunk. A query can match on any of them, improving retrieval coverage for questions that come at the content from different angles.
Content Embedding
Embeds the raw chunk text. Matches queries that ask directly about the content.
Summary Embedding
Embeds an LLM-generated summary of the chunk. Matches broad or thematic queries that wouldn't align with raw text.
Question Embedding
Embeds hypothetical questions this chunk could answer (ColBERT-style). Directly matches question-shaped queries.
All three embeddings point back to the same chunk. This app stores one dense embedding per chunk — multi-vector indexing is how production systems extend this when a single embedding doesn't cover enough query shapes.
Step 3 — Query Understanding, Hybrid Retrieval & Reranking
Before retrieval, the raw user question is transformed to maximise retrieval quality. Then hybrid search runs — identical to Hybrid RAG — but followed by a cross-encoder reranker that scores every candidate for precision.
Query Rewriting
An LLM rewrites the raw question to be more retrieval-friendly — more specific, precise, and aligned with how document text is phrased.
Original: "How does attention work?"
Rewritten: "Explain the self-attention mechanism in transformer models and how it computes token relationships"
HyDE — Hypothetical Document Embeddings
An LLM generates a hypothetical answer to the question. That answer (not the original question) is embedded for retrieval. A plausible answer is semantically closer to real stored chunks than a short question is.
Question: "How does self-attention work?"
HyDE answer: "Self-attention computes a weighted sum of values by comparing a query vector..."
→ embed this hypothetical answer → use for vector search
Query Expansion & Multi-Query
A single query only covers one angle. Different chunks in a document use different phrasing for the same idea — so one query will miss anything worded differently. The fix: generate multiple paraphrases and sub-questions from the original, run each through retrieval independently, then merge all results via RRF.
Example
Original: "What are the side effects of ibuprofen?"
Generated versionWhy it helps
"What are the side effects of ibuprofen?"original query
"What are the risks of taking ibuprofen?"synonym — "risks" instead of "side effects"
"Can ibuprofen cause stomach problems?"sub-question — targets one specific effect
"Is ibuprofen safe for long-term use?"sub-question — different angle
"ibuprofen adverse reactions"keyword phrasing — how a medical doc would word it
Each version runs retrieval independently. A chunk saying "adverse reactions include gastrointestinal bleeding" scores low against "side effects" but high against "adverse reactions" or "stomach problems" — query expansion ensures it gets found regardless of phrasing.
After hybrid retrieval + RRF, the candidate pool goes to the cross-encoder:

Unlike the bi-encoder used during indexing (which encodes query and chunks independently), the cross-encoder reads both as a single input — letting transformer attention directly compare their tokens. Slower but far more accurate. Only the top-scored chunks (e.g. C and A) proceed to generation.
This App
Reranking is done by the LLM (Claude). Claude is prompted to score each candidate chunk for relevance against the query and return a ranked order. This is why the demo results show "4. LLM Re-ranking" — no separate reranker model is needed, Claude already handles it.
Real-World Production
Reranking is done by a dedicated cross-encoder model — such as Cohere Rerank, BGE Reranker, or Jina Reranker. These are small, purpose-built models that are faster and cheaper per call than an LLM, and purpose-trained specifically for relevance scoring.
Step 4 — Context Compression & Generation
Before the LLM sees anything, a post-processing pass maximises signal and minimises token waste. The prompt is structured with source citations. A final validation step checks the answer for faithfulness against the retrieved evidence.
Deduplication
Remove near-duplicate chunks that would waste context window tokens on repeated information.
Summarisation
Extract only the relevant sentences from each chunk rather than passing the full chunk text.
Reordering
Place most-relevant chunks at the start and end of the context — LLMs attend more strongly to these positions.

When to Use It
  • Large, complex documents with mixed content — tables, images, varied formats
  • Users ask vague, multi-part, or domain-specific questions
  • Answer accuracy and reliability are non-negotiable
  • Production systems with real users in high-stakes environments
When Not to Use It
  • Building a quick prototype — each step adds latency and cost
  • Small, clean, well-structured document sets
  • Infrastructure budget is limited
  • Hybrid RAG has already been tested and proven sufficient
Strengths
  • Handles complex, ambiguous, and multi-part questions effectively
  • Metadata traceability — every answer can be traced to a source, page, and section
  • Reranking ensures only the most relevant context reaches the LLM
  • Answer validation catches hallucinations before they reach the user
Limitations
  • Significantly more complex to build, tune, and maintain
  • Higher latency — multiple LLM calls for rewriting, reranking, and validation
  • Greater infrastructure cost — reranker models, multi-vector storage
  • More failure points — each step can introduce its own errors
Key Concepts
HyDE — Hypothetical Document Embeddings
Instead of embedding the raw question, the LLM first generates a hypothetical answer to the question. That hypothetical is then embedded and used for retrieval. The idea: a fake but plausible answer is semantically closer to the real stored chunks than a short question would be.
Hybrid Search
Combines dense retrieval (vector similarity — good at semantic meaning) with sparse retrieval (keyword/BM25 — good at exact terms, names, codes). Results from both are fused using Reciprocal Rank Fusion (RRF), giving the best of both worlds. This app implements hybrid search in both the Advanced and Hybrid RAG endpoints.
Cross-Encoder Reranking
The bi-encoder used during indexing encodes query and chunks independently — fast but less accurate. A cross-encoder takes the query and each candidate chunk together as a single input and outputs a precise relevance score. Slower but far more accurate. Common models: Cohere Rerank, BGE Reranker, Jina Reranker, Cross-Encoder models.
Sliding Window & Parent-Child Chunking
Sliding window chunking uses overlap between consecutive chunks so no idea is cut at a boundary. Parent-child chunking stores small chunks for precise retrieval but also keeps a reference to the larger parent chunk — when a small chunk is retrieved, the full parent is sent to the LLM for richer context.
Reciprocal Rank Fusion (RRF)
A score-free fusion algorithm that merges multiple ranked lists. Each document gets a score of 1 / (k + rank) from each list, and the scores are summed. The constant k=60 is the standard value. This app uses RRF in both the Advanced and Hybrid RAG endpoints.
