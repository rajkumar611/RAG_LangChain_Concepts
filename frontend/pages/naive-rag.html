STRATEGY 1 OF 5 Naive RAG

The simplest and most direct form of RAG. It establishes the baseline — embed the query, find the closest chunks, hand them to the model. No rewriting, no re-ranking, no fusion.

How It Works
Step 1 — Document Ingestion & Chunking
Raw text is extracted from the document with minimal processing — footers, page numbers, and boilerplate remain as-is. The document is then split using a fixed character limit with no overlap.

What is stored: Each chunk is saved as a plain string in a Python list. No document name, source filename, page number, or section heading is attached — no metadata of any kind. The only way to locate a chunk is by its position in the list. Real-world vector databases store at minimum {"source": "report.pdf", "chunk": 3} alongside each chunk, enabling source citation and filtered retrieval. This implementation stores none of that.

What the LLM actually receives: At generation time, even the list position and similarity score are dropped before the prompt is built. The model sees only the raw chunk texts and your question.
Input text:
"Self-attention allows transformers to model relationships between tokens. It replaces recurrence used in RNNs. Training requires large datasets."

Fixed-size chunks:
Chunk 1: "Self-attention allows transformers to model relationships between tokens."
Chunk 2: "It replaces recurrence used in RNNs."
Chunk 3: "Training requires large datasets."
Key limitations of fixed-size chunking
  • Chunks may break in the middle of a sentence
  • Unrelated ideas can be grouped if they happen to be nearby
  • No overlap — context that spans two chunks may never be fully retrieved
  • No semantic awareness — topic shifts are invisible to the chunker
Step 2 — Embedding & Indexing
Each chunk is encoded into a single embedding vector representing its full content. These vectors are appended to a flat NumPy matrix (DOC_EMBS) — this matrix is the index. Only the vector and the raw text are stored; no additional signals, weights, or positional information.

In this app the index lives in memory and is lost on restart. In production the same vectors would be written to a dedicated vector database — such as Pinecone, Chroma, FAISS, or Weaviate — which adds persistence, scalability, and fast Approximate Nearest Neighbor (ANN) search without scanning every vector.
Chunk:
"Self-attention allows transformers to model relationships between tokens."

Embedding (384-dim):
→ [0.21, -0.88, 0.77, 0.12, -0.44, ...]

Model used: all-MiniLM-L6-v2 (same model for chunks and queries — downloaded from Hugging Face on first run, ~80 MB, cached locally at ~/.cache/huggingface/hub/. Pre-baked into the Docker image so the first container startup is instant.)
Step 3 — Query & Retrieval
The user's raw question is used directly — no rewriting, decomposition, or expansion. It is embedded using the same model and compared against all chunk vectors via cosine similarity (computed as a NumPy dot product). The top-K highest-scoring chunks are returned.
Cosine Similarity
cos(θ) = A · B ‖A‖ · ‖B‖
A = query vector  ·  B = chunk vector
Score → 1.0 (identical direction)  ·  0.0 (orthogonal)
Vector Angle
Smaller θ → higher similarity
query_vector vs chunk_1_vector → similarity: 0.91
query_vector vs chunk_2_vector → similarity: 0.74
query_vector vs chunk_3_vector → similarity: 0.43
query_vector vs chunk_4_vector → similarity: 0.21

Top 3 returned: chunk_1, chunk_2, chunk_3
Step 4 — Generation
The top-K chunks are concatenated with the user query and sent directly to the LLM. No filtering, deduplication, compression, or reordering happens first.

No answer validation is performed. Hallucinations — where the model generates plausible but unsupported claims — pass through silently.
Common Failure Cases
Exact match queries
Searching for an error code, product ID, or specific name — if the query wording doesn't match the chunk's embedding direction closely, the right chunk won't be in the top-K.
Multi-hop reasoning
Questions that require combining information from two separate sections of a document — there is no graph expansion or iterative search to bridge the gap.
Long docs with mixed topics
Fixed-size chunks from a long document mix different topics — unrelated chunks score higher than expected and pollute the context window.
Ambiguous queries
Without query rewriting or decomposition, vague questions retrieve whatever is semantically closest — often the wrong content.

When to Use It
  • Documents are clean, well-structured, and consistent
  • Users ask simple, direct, single-part questions
  • Building a quick prototype or proof of concept
  • Budget and infrastructure are limited
  • Response speed matters more than retrieval precision
When Not to Use It
  • Documents contain tables, mixed formats, or inconsistent terminology
  • Users ask complex or multi-part questions
  • High-stakes domains — legal, medical, or financial — where accuracy is critical
  • Large, noisy corpus with a lot of irrelevant content mixed in
Strengths
  • Minimal moving parts — easy to implement, debug, and explain
  • Single embedding pass + one vector search — very fast end-to-end
  • No external services or complex infrastructure needed
  • Ideal performance baseline before layering on more advanced techniques
Limitations
  • No metadata means retrieved chunks have no traceability to source or page
  • Boilerplate and noise from the document go straight to the model
  • No deduplication — the same content can appear multiple times in the context
  • No answer validation — hallucinations pass through silently
Key Concepts
Embeddings
A vector representation of text that captures semantic meaning. Similar sentences produce vectors that are close together in high-dimensional space. This app uses all-MiniLM-L6-v2 — a lightweight 384-dimension model from Sentence Transformers.
Cosine Similarity
Measures the angle between two vectors. A score of 1.0 means identical direction (most similar); 0.0 means orthogonal (no relation). Because embeddings are L2-normalised to unit length, cosine similarity reduces to a plain dot product. query_emb @ DOC_EMBS.T is a single matrix multiply: query_emb is shape (1 × 384) and DOC_EMBS is (n_chunks × 384); transposing it to (384 × n_chunks) and multiplying yields one similarity score per chunk in a single NumPy call. No approximate index is built — every query is compared against every chunk (brute-force linear scan), which is fast at small corpus sizes but would not scale to millions of chunks.
Fixed-Size Chunking
Documents are split by a fixed character limit with no overlap. Chunks are stored as plain strings with only an implicit integer index — no document name, page number, or section heading. This is the simplest chunking strategy — fast and predictable, but boundaries can fall mid-sentence, and without overlap, context that spans two chunks may never be fully retrieved.
