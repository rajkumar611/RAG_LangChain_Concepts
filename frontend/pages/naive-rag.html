STRATEGY 1 OF 5 Naive RAG

The simplest and most direct form of RAG. It establishes the baseline — embed the query, find the closest chunks, hand them to the model. No rewriting, no re-ranking, no fusion.

How It Works
Step 1 — Document Ingestion
Raw text is extracted from the document with no structural separation and no cleaning. Footers, page numbers, and boilerplate text remain as-is. The document is then split using a fixed character limit with no overlap — no metadata such as document name, page number, or section heading is stored alongside each chunk.
Step 2 — Indexing
Each chunk is encoded into a single embedding vector representing its full content. These vectors are appended to a flat NumPy matrix (DOC_EMBS) — this matrix is the index. Only the vector and the raw text are stored alongside each entry; no additional signals, weights, or positional information.

In this app the index lives in memory and is lost on restart. In production Naive RAG the same vectors would be written to a dedicated vector database — such as Pinecone, Chroma, FAISS, or Weaviate — which adds persistence, scalability, and fast approximate nearest-neighbour (ANN) search without scanning every vector.
Step 3 — Query & Retrieval
The user's raw question is used directly — no rewriting, decomposition, or expansion. It is embedded using the same model and compared against all chunk vectors via cosine similarity (computed as a NumPy dot product). The top-K scoring chunks are returned. No re-ranking is applied.
Cosine Similarity
cos(θ) = A · B ‖A‖ · ‖B‖
A = query vector  ·  B = chunk vector
Score → 1.0 (identical direction)  ·  0.0 (orthogonal)
Vector Angle
Smaller θ → higher similarity
Step 4 — Generation
All retrieved chunks are passed to the LLM as-is, without any filtering, deduplication, compression, or reordering. The prompt is unstructured, and no validation is performed on the final answer.
Full Pipeline — Step by Step
Ingestion Phase (Offline)
Step 1 — Document Ingestion & Basic Cleaning
Raw documents are loaded and text is extracted with minimal processing. Unlike Advanced RAG, there is no OCR pipeline, no structural separation of tables vs text, and no heavy cleaning pass.
Input Sources
  • PDFs
  • Web pages
  • Notes & internal docs
What gets skipped
  • No OCR for scanned docs
  • No footer / boilerplate removal
  • No structural parsing
  • Noise goes straight through
Step 2 — Fixed-Size Chunking
Text is split into equal-sized blocks by token or character count. There is no awareness of sentence or paragraph boundaries — the split happens at the character limit, wherever that falls.
Input text:
"Self-attention allows transformers to model relationships between tokens. It replaces recurrence used in RNNs. Training requires large datasets."

Fixed-size output (3 chunks):
Chunk 1: "Self-attention allows transformers to model relationships between tokens."
Chunk 2: "It replaces recurrence used in RNNs."
Chunk 3: "Training requires large datasets."
Key limitations of fixed-size chunking
  • Chunks may break in the middle of a sentence
  • Unrelated ideas can be grouped in the same chunk if they happen to be nearby
  • No overlap — context that spans two chunks may never be fully retrieved
  • No semantic awareness — topic shifts are invisible to the chunker
Step 3 — Embedding Generation
Each chunk is converted into a single embedding vector by the embedding model. This vector captures the semantic meaning of the chunk in high-dimensional space.
Chunk:
"Self-attention allows transformers to model relationships between tokens."

Embedding (384-dim):
→ [0.21, -0.88, 0.77, 0.12, -0.44, ...]

Model used: all-MiniLM-L6-v2 (same model for chunks and queries)
Step 4 — Single Storage: Vector DB Only
This is the biggest structural difference from Hybrid and Advanced RAG. Naive RAG uses one storage system — the vector database. There is no keyword index, no inverted index, no BM25.
Vector DB (used)
Stores chunk text + embedding vector. Enables cosine similarity search.
chunk → [0.21, -0.88, 0.77, ...]
→ Pinecone / Weaviate / NumPy matrix
Keyword Index (not used)
No inverted index is built. Exact keyword search is not available.
  • No BM25
  • No TF-IDF index
  • No inverted index
Typical Naive RAG metadata:
{
  "source": "doc.pdf",
  "chunk_id": 12
}

No section, topic, year, or page — just a pointer to the source file.
Query Time Phase (Online)
Step 5 — User Query & Query Embedding
The raw user question is used as-is — no rewriting, no HyDE, no decomposition into sub-questions. It is simply embedded using the same model that encoded the chunks.
Query: "How does self-attention work?"

Query vector: [0.18, -0.91, 0.74, ...]
What gets skipped at query time
  • No query rewriting for better retrieval alignment
  • No HyDE (hypothetical answer generation)
  • No query expansion or multi-query generation
Step 6 — Vector Similarity Search Only
The query vector is compared against every stored chunk vector using cosine similarity. The top-K highest-scoring chunks are returned. This is the only retrieval operation in Naive RAG.
query_vector vs chunk_1_vector → similarity: 0.91
query_vector vs chunk_2_vector → similarity: 0.74
query_vector vs chunk_3_vector → similarity: 0.43
query_vector vs chunk_4_vector → similarity: 0.21

Top 3 returned: chunk_1, chunk_2, chunk_3
No hybrid retrieval, no fusion, no reranking
  • No BM25 / keyword search
  • No RRF fusion
  • No cross-encoder reranking
  • Top-K similarity score is the final ranking — no further quality check
This means exact-match queries (error codes, product IDs, named entities) often fail — the vector space may not bring the right chunk to the top if the query wording differs from the chunk wording.
Step 7 — Prompt Construction & LLM Generation
The top-K chunks are concatenated with the user query and sent directly to the LLM. No filtering, deduplication, compression, or reordering happens first.
[System Prompt]
+
[Chunk 1] "Self-attention allows transformers to model..."
[Chunk 2] "It replaces recurrence used in RNNs."
[Chunk 3] "Training requires large datasets."
+
[User Query] "How does self-attention work?"

→ LLM generates answer
No answer validation is performed. Hallucinations — where the model generates plausible but unsupported claims — pass through silently.
Common Failure Cases
Exact match queries
Searching for an error code, product ID, or specific name — if the query wording doesn't match the chunk's embedding direction closely, the right chunk won't be in the top-K.
Multi-hop reasoning
Questions that require combining information from two separate sections of a document — Naive RAG has no graph expansion or iterative search.
Long docs with mixed topics
Fixed-size chunks from a long document mix different topics — unrelated chunks score higher than expected and pollute the context.
Ambiguous queries
Without query rewriting or decomposition, vague questions retrieve whatever is semantically closest — often the wrong content.

When to Use It
  • Documents are clean, well-structured, and consistent
  • Users ask simple, direct, single-part questions
  • Building a quick prototype or proof of concept
  • Budget and infrastructure are limited
  • Response speed matters more than retrieval precision
When Not to Use It
  • Documents contain tables, mixed formats, or inconsistent terminology
  • Users ask complex or multi-part questions
  • High-stakes domains — legal, medical, or financial — where accuracy is critical
  • Large, noisy corpus with a lot of irrelevant content mixed in
Strengths
  • Minimal moving parts — easy to implement, debug, and explain
  • Single embedding pass + one vector search — very fast end-to-end
  • No external services or complex infrastructure needed
  • Ideal performance baseline before layering on Advanced RAG techniques
Limitations
  • No metadata means retrieved chunks have no traceability to source or page
  • Boilerplate and noise from the document go straight to the model
  • No deduplication — the same content can appear multiple times in the context
  • No answer validation — hallucinations pass through silently
Key Concepts
Embeddings
A vector representation of text that captures semantic meaning. Similar sentences produce vectors that are close together in high-dimensional space. This app uses all-MiniLM-L6-v2 — a lightweight 384-dimension model from Sentence Transformers.
Cosine Similarity
Measures the angle between two vectors. A score of 1.0 means identical direction (most similar); 0.0 means no relation. This app computes it as a NumPy dot product — query_emb @ DOC_EMBS.T — across all stored chunk vectors at once. No FAISS index is used here; FAISS only appears in the LangChain RAG demo.
Fixed-Size Chunking
Naive RAG splits documents by a fixed character limit with no overlap and stores no metadata alongside each chunk. This is the simplest chunking strategy — it is fast and predictable, but boundaries can fall mid-sentence, and without overlap, context that spans two chunks may never be fully retrieved.
Reranking Not used in Naive RAG
Rerankers are small specialised models used to re-evaluate and reorder chunks that were already retrieved by vector search. Naive RAG skips this step entirely — whatever cosine similarity returns goes straight to the LLM. Understanding reranking explains exactly where Naive RAG falls short.
Pipeline with Reranking
User Query
→ Vector search (cosine similarity)
→ Reranker re-evaluates retrieved chunks
→ Orders by best relevance match
→ Best chunks augmented into prompt
→ Sent to LLM
