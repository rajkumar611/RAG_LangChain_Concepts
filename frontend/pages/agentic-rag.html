How It Works
Step 1 — Tool-Calling Setup
The LLM is given a search tool and instructed to use it to find information. It decides when to call the tool and what query to use. Unlike fixed pipelines, the agent controls both the timing and content of every retrieval step based on what it needs to answer the question.
Step 2 — Iterative Retrieval
After each tool call, the LLM evaluates whether it has enough information. If not, it issues another search with a refined or different query. Up to max_search_rounds iterations. Each round can use a completely different query strategy based on what was learned from the previous round's results.
Step 3 — Multi-hop Reasoning
For complex questions requiring information from multiple sections, the agent naturally chains tool calls without explicit programming. The LLM decides to search for topic A, then based on what it finds, searches for related topic B — making connections a fixed pipeline could never anticipate.
Step 4 — Answer Generation
Once satisfied with its collected evidence, the agent generates a final answer grounded in all retrieved content. The answer synthesises information across multiple retrieval rounds, producing responses that can span topics and draw connections a single-pass retrieval would miss.
Full Pipeline — Step by Step
Ingestion Phase (Offline)
Step 1 — Document Ingestion
Same sources as other RAG systems. The difference in Agentic RAG is not how documents are ingested — it is how they are retrieved at query time.
Example document chunk:
"Company policy states that refund requests must be processed within 7 days."
Step 2 — Chunking & Metadata Enrichment
Agentic RAG uses semantic or sentence-aware chunking (same as Advanced RAG) and attaches rich metadata — because the agent may use metadata fields as routing signals when selecting tools.
{
  "source": "refund_policy.pdf",
  "section": "Return & Refund",
  "topic": "policy",
  "chunk_id": 17
}
Step 3 — Multi-System Storage: A Toolbox of Retrievers
Unlike Naive RAG (vector only) or even Hybrid RAG (vector + BM25), Agentic RAG does not commit to one retrieval method. It maintains a toolbox — the agent picks the right tool for each sub-question.
Vector DB
Semantic similarity search.
Pinecone · Weaviate · Milvus
Keyword Index
Lexical / exact match.
Elasticsearch · BM25
Graph DB (optional)
Relationship traversal.
Neo4j · entity links
External APIs (CRM systems, ticketing tools, SQL databases) can also be registered as tools — the agent calls them the same way it calls a retriever.
Query Time Phase — Agent Loop (Online)
Step 4 — User Query Arrives & Agent Plans
This is the core difference from all other RAG types. Instead of immediately retrieving documents, an LLM acts as a controller that first analyses the question and builds a retrieval plan.
User query: "Can I get a refund if I cancel after 10 days?"

Agent reasoning:
This question involves:
- policy rules (factual retrieval needed)
- a time condition (10 days vs policy limit)
- requires comparing user situation to policy

Plan:
Step 1: Search refund policy documents
Step 2: Check time constraints section
Step 3: Validate eligibility condition
Step 4: Synthesise answer
No other RAG type does this. Naive, Hybrid, and Advanced RAG go straight from query to retrieval. The agent thinks before it retrieves.
Step 5 — Tool Selection
The agent does not always use the same retrieval method. It selects the tool best suited to the sub-question at hand.
Vector search
Semantic similarity — best for conceptual questions ("explain...", "how does...", "what is...")
Keyword search
Exact match — best for policy lookups, error codes, named entities ("7 days", "clause 4.2")
Graph search
Relationship reasoning — "what does refund policy link to?" → Terms & Conditions → constraint lookup
External API
CRM, ticketing systems, SQL databases — any registered callable function
Step 6 — First Retrieval Call + Reflection Loop
The agent issues its first tool call, reads the results, and reflects: do I have enough information to answer? If not, it refines the query and calls again. This is not a pipeline — it is a loop.
Think → Act → Observe cycle
THINK: "Need to find the refund time limit."
↓
ACT: vector_search("refund policy cancellation days")
↓
OBSERVE: Returns Chunk A (refund rules) + Chunk B (cancellation section)
↓
THINK: "I have the policy. Do I know the 10-day condition explicitly?"
↓ → need more info, loop again
ACT: keyword_search("10 days refund eligibility")
↓
OBSERVE: Returns exact clause: "refunds only within 7 days"
↓
THINK: "I now have enough. Ready to answer." → exit loop
This app bounds the loop to max_search_rounds iterations (default: 2) to control cost and latency. Without a cap, a poorly-specified question could cause the agent to loop indefinitely.
Step 7 — Context Assembly & Final Reasoning
Unlike fixed pipelines where retrieved chunks go directly to the LLM, the agent decides what context is useful. It may discard irrelevant chunks, re-rank internally, or summarise intermediate findings before producing the final answer.
Agent final reasoning:
User cancelled after 10 days.
Policy allows refunds only within 7 days.
10 > 7 → condition NOT met.
→ Not eligible for refund.
A self-verification step may follow — the agent checks whether it used the correct policy clause, whether the answer is grounded, and whether another retrieval step is needed before sending the final response.
Self-verification checks
  • "Did I use the correct policy?"
  • "Is the answer grounded in retrieved text?"
  • "Do I need another retrieval step?"
Agent-controlled context
  • Discards irrelevant chunks
  • Re-ranks internally by usefulness
  • Summarises intermediate results

Agentic RAG vs Other RAG Types
Type Retrieval Behaviour Decision-Making
Naive RAG Single vector search None — direct similarity lookup
Hybrid RAG Vector + BM25, always both Fixed — always runs both paths
Advanced RAG Multi-stage fixed pipeline Fixed — same steps every time
Graph RAG Seed + graph traversal Fixed — BFS hops are predetermined

When to Use It
  • Multi-hop questions that require combining evidence from multiple sections
  • Open-ended research questions with unpredictable scope
  • When you don't know in advance how many retrieval steps are needed
When Not to Use It
  • Simple factual lookups — Agentic RAG is overkill; one retrieval pass is enough
  • Latency-sensitive applications — each tool call adds response time
  • When queries are always predictable — Advanced RAG is cheaper and faster
Strengths
  • Naturally handles multi-hop questions without manual pipeline design
  • Self-directed — no manual pipeline tuning needed
  • Can refine its search strategy based on intermediate results
Limitations
  • Non-deterministic — the same question can take different retrieval paths
  • Harder to debug and trace than a fixed pipeline
  • Cost scales with the number of tool calls made
  • Can loop or over-retrieve if not bounded by a round limit
Key Concepts
Tool Calling
The LLM is given a function signature (a search tool) and decides when and how to call it. The function runs on the server, and its result is returned to the LLM as additional context. The LLM never executes code directly — it only requests the call.
Agent Loop
The core pattern is think → act → observe → think. The agent thinks about what it needs, calls a tool (act), reads the result (observe), and thinks again about whether it has enough to answer. This loop repeats until the agent is satisfied or hits the round limit.
max_search_rounds
This app limits the number of iterations via config (default 2) to bound cost and latency. Without a cap, a poorly-specified question could cause the agent to loop indefinitely, burning API budget. The limit is a safety valve — most well-formed questions resolve in 1–2 rounds.
Implementation in This App
This app uses claude-sonnet-4-6 directly via the Anthropic SDK. The agent is implemented using Anthropic's native tool-calling API — no LangChain or framework is used in the RAG module.
